# Service orchestrator

ORK — the Orchestration Kernel, distributed as `@tetherto/mdk-ork` — is the pull-only kernel that owns the device registry, routes commands to workers, and aggregates telemetry. This page documents the baseline shape from the workers' perspective. The orchestrator package itself (`backend/core/ork/`) will own a per-artefact `USAGE.md` once that pattern lands across `core/`.

## The two entry points

### `getOrk(opts?)` — preferred

One call, autogenerated topic, sensible cadences, attached `.topic` for downstream `startWorker` calls. Use this in any new SDK code unless you need to control the discovery topic, store directory, or gateway configuration explicitly.

```js
const { getOrk } = require('@tetherto/mdk')
const ork = await getOrk()
// ork.topic — auto-generated DHT topic (hex)
// ork.getPublicKey() — HRPC public key
```

See [`backend/core/examples/miners/mdk.client.miner.js`](../../core/examples/miners/mdk.client.miner.js) for the simplest end-to-end use.

### `new OrkManager({}, { storeDir, root })` — low-level

Use when you need a specific DHT topic, a fixed on-disk store directory, or explicit gateway config. The full surface is documented in [`backend/core/ork/index.js`](../../core/ork/index.js) (factory at top of file).

```js
const { OrkManager } = require('@tetherto/mdk-ork')
const ork = new OrkManager({}, { storeDir, root })
await ork.init()
await ork.start()
```

See [`backend/core/examples/ork/mdk.client.ork.js`](../../core/examples/ork/mdk.client.ork.js) for the full bootstrap including config file generation and SIGINT handling.

## How workers connect

Workers join the same DHT topic as ORK. ORK pulls — workers never push. Once a worker announces on the topic:

1. ORK discovers it via DHT.
2. ORK opens an HRPC channel.
3. ORK begins the configured pull cadences (`telemetryPullMs`, `statePullMs`, `healthPingMs`).
4. The worker answers each pull with current telemetry or state from its registered things.

Commands flow back along the same channel: ORK → worker → device. The worker translates the MDK Protocol envelope into the device's native API call.

For multi-process discovery (worker and ORK in separate Node processes), see [`backend/core/examples/mdk-e2e/dht-worker.js`](../../core/examples/mdk-e2e/dht-worker.js) and [`dht-ork.js`](../../core/examples/mdk-e2e/dht-ork.js).

## Storage and lifecycle

Each ORK process writes to a `storeDir` containing a Hyperbee for the registry. Two ORK instances pointed at the same `storeDir` will collide on file locks. The bootstrap helpers default to `os.tmpdir() + '/mdk'` to keep examples isolated; production deployments configure explicit paths.

Lifecycle:

```
new OrkManager() -> .init() -> .start() -> ... -> .stop()
```

`getOrk()` runs `.init()` and `.start()` for you. All examples install a `SIGINT` handler that calls `.stop()` before exiting.

## Configuration cadences

ORK pulls on three schedules, configurable via `getOrk({ cadences: ... })` or the low-level config:

| Cadence | Default | Purpose |
| --- | --- | --- |
| `telemetryPullMs` | 10000 | How often to pull live metrics from each worker. |
| `statePullMs` | 60000 | How often to pull device state (slower than telemetry). |
| `healthPingMs` | 5000 | How often to ping workers for liveness. |

Tune downward for tighter dashboards; tune upward to reduce DHT traffic.

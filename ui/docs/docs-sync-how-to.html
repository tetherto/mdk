MDK · Documentation

Keep the docs in sync
in one command.

When the MDK codebase changes, the reference pages — every component, hook, store, utility and font — regenerate themselves. You run one command, read a short report, and write the words only a human can. No code, no nav files, no copy-pasting.

Everything here runs in the docs folder, mdk-docs. It reads your code from the mdk-prv folder next to it, but you never run doc commands inside mdk-prv.

mdk-docs — zsh
$npm run docs:generate
✔ synced prose from the monorepo
✔ regenerated reference dataset (latest release)
✔ +3 pages created · 1 removed · nav updated
✔ report → src/data/sync-report/index.html

Once per machine

Point it at your code checkout

The generator reads from your local copy of the MDK monorepo. Tell it where that lives — you only do this once. It documents whatever branch you have checked out, so you don't need to match any particular branch name here.

Copy docs-sync.config.local.example.yaml to docs-sync.config.local.yaml and set the path: 

sources:
  local:
    localPathToRepo: /path/to/your/mdk-prv   # your code checkout

Before every run

Pull the latest code first

The docs describe the code you have checked out right now. Both the MDK and the docs repos are forks, so before you generate, make sure your MDK checkout has the latest develop pulled in from upstream — otherwise the docs will quietly describe stale code.

In your mdk-prv folder, get the newest code:

git checkout develop
git pull upstream develop   # upstream = the original tetherto repo

The generate command prints which branch and commit it used for both repos, so you can always confirm what the docs were built from.

Every time the code changes

The four-step loop

01

Run the generator

One command, run in the mdk-docs folder. Regenerates the reference for every public part of MDK from your current code checkout, rebuilds the navigation, and writes you a report. Safe to run any time — it only touches generated pages.

$cd mdk-docs && npm run docs:generate
02

Open the report

A single page that tells you exactly what changed and where to spend your time — new pages, refreshed pages, deleted pages, and a prioritised “where to focus” list. It's written inside the mdk-docs folder — just double-click it.

mdk-docs/src/data/sync-report/index.html
03

Write the intro prose

The props, signatures, usage and examples are already on the page. For a new component or hook, add the part only you can write — the “why” and “when to use it” around the auto-rendered block.

04

Preview it

See the site exactly as readers will, with your prose and the fresh reference side by side.

$npm run dev

Reading the report

Three things to look for

New

A page was created. Write the intro prose around the reference block.

Changed

The code changed and the page refreshed itself. Skim it to check your surrounding words still fit.

Removed

Something was deleted from the code, so its page is gone. Fix any links the report flags.

Changing what a page says

Edit the source, not the generated page

Generated pages are built from the MDK source code, so edits made directly on them are overwritten next run. To change what a generated page says, edit the source in the mdk-prv repo, then run npm run docs:generate again — your change flows through.

Where each piece of a generated page comes from:

  • →The summary at the top — the component/hook’s doc comment (JSDoc) in its source file.
  • →The “Usage” prose — that item’s USAGE.md next to its source.
  • →The examples — its *.example.tsx files.
  • →The props / signatures — read straight from the TypeScript types.

Not sure where a file lives? Ask a developer — then re-run the one command and check the report.

Leave it alone

What you never have to touch

  • →Generated pages — anything marked mdk:generated is overwritten on the next sync. Edits there are lost; write in your own pages instead.
  • →The navigation — new pages slot into the sidebar on their own.
  • →The plumbing — data files, renderers and registration are all handled for you.

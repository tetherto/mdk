# Extending the docs pipeline to the backend

_How to make the auto-generated docs cover **all** of MDK — the backend
packages too, not just the UI._

Today the one command (`npm run docs:generate`, run in the `mdk-docs` repo)
documents the **UI** side of MDK: components, hooks, stores, query helpers,
utilities and fonts. The backend (`backend/core/*`, `backend/workers/*`) is not
covered yet. This guide explains exactly what the backend has to do to join the
same pipeline, using the same mechanism the UI already uses — so a backend
engineer adds a manifest, and the existing one command starts producing backend
reference pages automatically.

You do **not** need a second pipeline. The pipeline is generic; it just needs a
new **input manifest** and a small amount of CLI wiring to render it.

---

## 1. How the UI pipeline works today (the pattern to copy)

Every documented UI package emits a **machine-readable manifest** at build time.
Nothing reads source code at doc time — the docs are built from these manifests.

| Package | Build step | Manifest | Generated by |
| --- | --- | --- | --- |
| `@tetherto/mdk-react-devkit` | `build` | `dist/registry.json` | export/JSDoc extraction |
| `@tetherto/mdk-react-adapter` | `build` | `dist/hooks.json` | export/JSDoc extraction |
| `@tetherto/mdk-ui-foundation` | `build:stores` | `dist/stores.json` | [`generate-stores.mts`](../packages/ui-foundation/scripts/generate-stores.mts) (ts-morph) |
| `@tetherto/mdk-fonts` | `build` | `package.json` + `dist/fonts/` | static assets |

The flow, all inside the MDK CLI (`@tetherto/mdk-ui-cli`):

```
build monorepo ─► manifests (registry/hooks/stores/fonts.json)
        │
        ▼
mdk-ui docs:generate ─► loads each manifest ─► buildDataset()
        │                                          │
        │                                          ▼
        │                              dataset → JSON + MDX pages + nav
        │                                          │
        ▼                                          ▼
  writes into mdk-docs/src/data/<version>/generated + content/…  + report
```

Two independent lanes make up the full docs:

- **Reference** (structured): the manifests above → dataset → pages. Owned by
  the CLI (`mdk-ui docs:generate`, which wraps `docs:build`).
- **Prose** (hand-written): Markdown in the monorepo tagged with a
  `docs@tether_slug` front-matter key, ported by the docs repo's
  `sync:docs` into the site's content tree.

The key idea: **a package earns a reference page by publishing a manifest that
matches a contract the CLI understands.** That's the only contract.

---

## 2. The manifest contract

A manifest is a single JSON file with a stable, versioned schema. Look at
`stores.json` as the reference implementation — it is the closest analogue to
what the backend needs (a set of named "things" with descriptions, categories,
signatures, and a source file):

```jsonc
{
  "version": "1.1.0",              // schema version (bump on breaking changes)
  "package": "@tetherto/mdk-ui-foundation",
  "stores": [
    {
      "name": "actionsStore",
      "category": "actions",
      "description": "Module-level singleton holding the queue of …",
      "state":   [{ "name": "pendingSubmissions", "signature": "PendingSubmissionAction[]" }],
      "actions": [{ "name": "enqueue", "signature": "(a: Action) => void" }],
      "factory": "createActionsStore",
      "file": "src/store/actions.ts"
    }
  ]
}
```

Rules that make it work with the pipeline:

1. **Deterministic.** Same source ⇒ byte-identical JSON (sort keys/arrays). The
   pipeline diffs manifests to report what changed; non-determinism creates
   noise and false "changed" flags.
2. **Public surface only.** Never emit anything private. The CLI runs a **leak
   guard** that aborts the whole build if forbidden strings (e.g. `mdk-prv`)
   appear in the output.
3. **Self-describing records.** Each record carries everything a page needs —
   name, category, description, signatures, and a `file` pointer — so the
   renderer never has to read source.
4. **Written to `dist/`** during the package's normal build, so a fresh build
   always refreshes it. (This is why `docs:generate` does a full build first;
   a stale manifest silently drops surface.)

## 3. What the backend has to do

### Step A — Decide the backend's "public surface"

Unlike the UI (components/hooks), the backend's documentable units are things
like:

- **Services / modules** exported by `backend/core/*` (e.g. `mdk`, `kernel`,
  `client`, `plugins`) — their public classes/functions and signatures.
- **Workers** in `backend/workers/*` (containers, miners, minerpools,
  power-meter, temperature) — their purpose, the RPC methods they expose, the
  events/topics they emit, and their config schema.
- **RPC / API methods** — name, params, return shape, description.
- **Config schemas** — the keys each package reads.

Pick the units and design **one manifest schema per kind** (e.g.
`backend-services`, `backend-workers`). Keep each record shaped like the
`stores.json` record: `{ name, category, description, methods[], events[],
config[], file }`.

### Step B — Emit the manifest at build time

Add a generator to each backend package, mirroring
[`generate-stores.mts`](../packages/ui-foundation/scripts/generate-stores.mts):

- The backend is JavaScript (`standard` lint), so extract the surface with
  **JSDoc + a static analyzer** (ts-morph works on JS with
  `allowJs`, or use `jsdoc`/`doctrine`, or hand-maintain a typed descriptor
  module the script serializes).
- Write `dist/backend.json` (or `dist/workers.json`).
- Wire it into the package build, e.g. add `"build:manifest": "node scripts/generate-backend.mjs"`
  and call it from the package's `build`. The backend workspaces build via
  `install-packages.sh`; make each documented package produce its manifest as
  part of that.

At this point the backend produces manifests but nothing consumes them yet.

### Step C — Teach the CLI to ingest the manifest

This is the wiring, and it mirrors the existing `stores` lane one-to-one. In
`ui/packages/cli/src`:

| File | Add (mirroring stores) |
| --- | --- |
| `registry-loader.ts` | `loadBackendManifest()` + `loadBackendManifestFromDir()` |
| `docs-data.ts` | a `DocsBackendRecord` type; accept `backend` in `buildDataset()`; emit `backend.json` (+ usage/example aux) from `datasetToFiles()`; include it in `computeDrift()` |
| `docs-scaffold.ts` | a scaffold category so backend records become MDX pages under a `content/docs/backend/…` tree, wired into the nav |
| `templates-docs/BackendDoc.tsx` | a renderer component (like the component/hook/store renderers) that reads a record and renders it; auto-installed + registered in `mdx-components.tsx` |
| `docs-report.ts` | count backend records in the doc-writer report |
| `commands/docs-build.ts` | resolve a `--backend-dir` (and package-name fallback), load the manifest, pass it to `buildDataset` |
| `commands/docs-generate.ts` | auto-resolve the backend package dirs (like `react-devkit` etc.) so the one command picks them up with no extra flags |
| `bin.ts` | expose `--backend-dir` on `docs:build` / `docs:generate` |

Because `docs:generate` already builds the whole monorepo and walks the package
dirs, once these dirs are resolved the one command covers the backend with **no
change to `mdk-docs`** beyond the renderer install (which is automatic).

### Step D — Port the backend prose

The backend already has `backend/core/docs/` and `backend/workers/docs/`. To get
those hand-written pages onto the site, tag each Markdown file's front-matter
with a `docs@tether_slug` (the same key the UI prose uses). `sync:docs` sweeps
the repo root (`pathInsideRepo: .`), so tagged backend Markdown is picked up
automatically — no config change needed.

---

## 4. Definition of done

- [ ] Each documented backend package writes a deterministic `dist/*.json`
      manifest during its build, public surface only.
- [ ] The CLI has a loader + record type + dataset wiring + scaffold category +
      renderer for the backend manifest (the table in Step C).
- [ ] `mdk-ui docs:generate` auto-resolves the backend package dirs.
- [ ] Backend Markdown carries `docs@tether_slug` tags.
- [ ] Running `npm run docs:generate` in `mdk-docs` produces backend reference
      pages + prose, and the report counts them — with the **same one command**.

---

## 5. Why this shape

- **One pipeline, many inputs.** The CLI already treats components, hooks and
  stores as interchangeable "record kinds." Backend is just another kind — no
  parallel tooling, no second command.
- **Manifests, not source, at doc time.** Keeps doc generation fast,
  deterministic, and safe (the leak guard sees only the manifest).
- **The build is the source of truth.** Because manifests are a build artifact,
  "the docs describe the code you have checked out" holds for the backend too.
